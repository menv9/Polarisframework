## 5. Riesgos Estructurales

### 5.1 Sesgo del builder

Diseñé el framework durante 20+ iteraciones. Cualquier persona en ese rol acumula **preferencias sutiles no cuestionadas**. Cosas que parecen obvias tras pensarlas 10 veces pueden ser arbitrarias para alguien externo.

**Mitigación:** **review externo obligatorio** antes de operar. Idealmente un quant senior que no haya participado en la conversación.

### 5.2 Régimen-dependence

El framework está calibrado mentalmente al régimen 2010-2024:
- QE → unwinding → inflación post-COVID → tipos altos
- USD relativamente estable como reserve currency
- Bancos centrales como driver dominante

Si el régimen cambia estructuralmente:
- De-dollarization material
- Fragmentación de bloques monetarios
- CBDC adoption masiva
- Cambio en correlación equity-bond

→ Los β cambiarán de signo / magnitud. El sistema podría fallar **silenciosamente** durante meses antes de detectarlo.

**Mitigación parcial:** §16 Robustez Temporal + Triage A/B/C. Pero ninguna defensa es absoluta.

### 5.3 Concentración de fuente de datos

FRED es gratuito pero está bajo gobierno US. yfinance depende de Yahoo. ECB SDW podría cambiar API.

Si una fuente crítica desaparece o cambia formato → disrupción no trivial.

### 5.4 Capacity ceiling no modelado

El framework no tiene cálculo explícito de **capacidad máxima** del sistema. ¿A partir de qué AUM tu propio trading mueve el mercado?
- EUR/USD: muy alta (millones sin impact)
- USD/MXN: media
- USD/TRY: baja (cientos de miles ya impactan)
- Cross-asset crypto: variable

Sin esta info, el escalado puede destruir el edge en cierto punto.

### 5.5 Single point of failure: el operador

Todo el sistema descansa sobre un único humano (tú). Si quedas incapacitado prolongadamente:
- Disaster Recovery §4.7 dice "parar el sistema" pero no resuelve la continuidad
- Knowledge transfer no formalizado en detalle
- No hay sucesor entrenado

Para retail solo, aceptable. Para capital significativo, **gap material**.
