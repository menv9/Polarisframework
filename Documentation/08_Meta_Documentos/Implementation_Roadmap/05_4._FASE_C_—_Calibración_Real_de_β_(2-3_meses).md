## 4. FASE C — Calibración Real de β (2-3 meses)

### Objetivo
Convertir los β placeholder en β estadísticamente validados. Esta es **la fase más crítica de todo el plan.**

### Mes 5: Backtest framework

```
SEMANA 17-18:
  ☐ Backtest engine básico:
    - Iteración día a día sin look-ahead
    - Point-in-time data (ALFRED para series con revisiones)
    - Cálculo de señales con datos disponibles en t
  ☐ Simulación de trades con costes
  ☐ Cálculo de P&L histórico

SEMANA 19-20:
  ☐ Walk-forward implementation:
    - Train 7 años, test 1 año
    - Slide ventana cada año
    - Sin overlap entre train y test
  ☐ Embargo y purging (López de Prado)
  ☐ Métricas IS vs OOS
```

### Mes 6: Calibración + Tests Anti-Espuria

```
SEMANA 21-22:
  ☐ Regresión OLS con statsmodels
  ☐ Cálculo de β iniciales calibrados
  ☐ Aplicar tests §17 Endo a cada driver:
    - Test 1: p-value < 0.05 (Bonferroni si aplica)
    - Test 2: justificación económica escrita por driver
    - Test 3: estabilidad β en dos subperíodos 5Y
    - Test 4: VIF < 5
    - Test 5: Sharpe OOS / Sharpe IS > 0.6

SEMANA 23-24:
  ☐ Lista final de drivers que pasan los 5 tests
  ☐ β calibrados guardados en YAML versionado
  ☐ Documentación: ¿qué drivers cayeron y por qué?
  ☐ Comparación métodos β: anual fija vs rolling 60m vs Kalman
  ☐ Elección de método para producción

EXPECTATIVA REALISTA:
  De 24 drivers candidatos → 10-14 sobrevivirán probablemente
  Algunos drivers favoritos (UMCSI, Building Permits) caerán
```

### Mes 7: Validación y métricas

```
SEMANA 25-26:
  ☐ Backtest formal sobre 15-20 años de histórico
  ☐ Walk-forward con últimos 3 años como OOS estricto
  ☐ Cálculo de métricas mínimas (§9.4 Endo):
    - Sharpe IS y OOS
    - Sortino, Calmar
    - Max DD, recovery time
    - Hit rate, profit factor
    - IC

SEMANA 27-28:
  ☐ Análisis de resultados:
    - ¿Cumple Sharpe OOS > 0.5?
    - ¿MDD < 25%?
    - ¿Hit rate > 50%?
    - ¿IC > 0.05?
  ☐ Si NO cumple: diagnosticar (drivers? señal? costes?)
  ☐ Si SÍ cumple: documentar oficialmente el modelo v0.2

EXPECTATIVA REALISTA:
  Primera iteración: probablemente NO cumple métricas mínimas
  Iterar: ajustar drivers, pesos, refinements (sin sobreajustar)
  Segunda o tercera iteración: debería cumplir si la lógica es correcta
```

### Decision gate C → D
```
PARA AVANZAR A FASE D:
  ✓ β calibrados con regresión real (no placeholders)
  ✓ Drivers han pasado los 5 tests anti-espuria
  ✓ Backtest formal con métricas mínimas cumplidas
  ✓ Documentación oficial de modelo v0.2

  CRÍTICO: si Sharpe OOS < 0.3 tras varias iteraciones:
    → El sistema NO funciona en su forma actual
    → Volver a Fase B (revisar lógica) o ABANDONAR proyecto
    → Mejor saberlo ahora que tras operar 6 meses
```
