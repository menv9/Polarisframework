## 14. Anti-Patterns a Evitar

```text
1. SOBRE-INGENIERÍA EN ARRANQUE
   Empezar con Kafka, microservices, Kubernetes para 1 usuario y 24 pares
   = procastinación disfrazada de arquitectura.
   → Empezar con scripts Python locales + Parquet. Escalar cuando duela.

2. NO USAR VERSION CONTROL
   "Lo subo a git cuando tenga algo bueno" = nunca.
   → git desde día 1. Commits pequeños y frecuentes.

3. SECRETS EN CÓDIGO
   API keys hardcoded para "no perder tiempo con .env" = data breach.
   → .env desde día 1, en .gitignore.

4. SIN TESTS
   "Después escribo tests cuando esté estable" = bugs en producción.
   → Tests para funciones que tocan dinero, mínimo 60% coverage.

5. ACOPLAMIENTO DE CAPAS
   Capa 1 importa funciones de Capa 2 importa Capa 3 = imposible cambiar nada.
   → Cada capa expone interfaz clara. Comunicación vía data, no función.

6. RECALIBRACIÓN MANUAL
   "Cada año recalibro a mano" = inconsistente, lento, propenso a error.
   → Pipeline automatizado de recalibración, con tests de aceptación.

7. NO MONITORIZAR
   "Si falla algo me entero al ver el P&L mensual" = pérdidas innecesarias.
   → Health checks + alertas automáticas críticas + dashboard.

8. CAMBIOS EN PRODUCCIÓN SIN BACKTEST
   "Tweakeo este β y veo qué pasa con money real" = ruina.
   → Backtest obligatorio + paper trading + sizing reducido.

9. UN SOLO BROKER PARA TODO
   Capa 1 FX y Capa 3 equities en mismo broker → riesgo concentrado.
   → IB unificada o brokers separados con transferencias claras.

10. NO LOGGING DE DECISIONES
    "Por qué abrí ese trade?" — 6 meses después, nadie sabe.
    → Loggear señal, score, regime state, motivo en cada trade.
```
